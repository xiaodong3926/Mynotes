## 1.从悬垂引用认识生命周期

- **悬垂引用**是指一个引用指向了已经被释放的内存区域，像这样:
```
{
    let r;  //声明变量r（此时类型未确定）

    {
        let x = 5;      //创建变量x，生命周期开始
        r = &x;         //r引用x，r的生命周期依赖于x
    }                   //x的生命周期结束，x被销毁

    println!("r: {}", r);  //尝试使用已失效的引用
}
```

- **借用检查**

Rust中有一个检查器，来检查代码是否符合借用的正确性:

```
{
    let r;                // ---------+-- 'a
                          //          |
    {                     //          |
        let x = 5;        // -+-- 'b  |
        r = &x;           //  |       |
    }                     // -+       |
                          //          |
    println!("r: {}", r); //          |
}                         // ---------+

```

 'a 与 ‘b 分别代表了 r 与 x 的生命周期，在编译期，Rust 会比较两个变量的生命周期，结果发现 `r` 明明拥有生命周期 `'a`，但是却引用了一个小得多的生命周期 `'b`，在这种情况下，编译器会认为我们的程序存在风险，因此拒绝运行

如果想要编译通过，也很简单，只要 `'b` 比 `'a` 大就好。总之，`x` 变量只要比 `r` 活得久，那么 `r` 就能随意引用 `x` 且不会存在危险：
```
{
    let x = 5;            // ----------+-- 'b
                          //           |
    let r = &x;           // --+-- 'a  |
                          //   |       |
    println!("r: {}", r); //   |       |
                          // --+       |
}                         // ----------+

```

## 2.函数中的生命周期

在存在多个引用时，编译器有时会无法自动推导生命周期，此时就需要我们手动去标注，通过为参数标注合适的生命周期来帮助编译器进行借用检查的分析:
```
fn main() {
    let string1 = String::from("abcd");
    let string2 = "xyz";

    let result = longest(string1.as_str(), string2);
    println!("The longest string is {}", result);
}

fn longest(x: &str, y: &str) -> &str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}
```

这段代码就是错的:原因是编译器无法知道该函数的返回值到底引用 x 还是 y ，**因为编译器需要知道这些，来确保函数调用后的引用生命周期分析** :由于编译器在**编译阶段**不知道 `if` 条件的结果，它无法保证返回的引用在 `println!` 执行时依然有效

## 3.生命周期标注语法

对与上面的代码，你需要使用生命周期参数来告诉编译器，输入参数和返回值的生命周期之间存在某种关联(标注并不改变引用的实际寿命，它只是一种契约)

生命周期的语法也颇为与众不同，以 `'` 开头，名称往往是一个单独的小写字母，大多数人都用 `'a` 来作为生命周期的名称。 如果是引用类型的参数，那么生命周期会位于引用符号 `&` 之后，并用一个空格来将生命周期和引用参数分隔开:
```
&i32        // 一个引用
&'a i32     // 具有显式生命周期的引用
&'a mut i32 // 具有显式生命周期的可变引用
```

为之前的那段代码的函数部分标注生命周期:
```
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}
```

- 和泛型一样，使用生命周期参数，需要先声明 `<'a>`
- `x`、`y` 和返回值至少活得和 `'a` 一样久（因为返回值要么是 `x`，要么是 `y`）
- 生命周期 `'a` 不代表生命周期等于 `'a`，而是大于等于`‘a`

`x: &'a str, y: &'a str`  意味着找出一个它们共同存活的时间段，并把这个时间段命名为 'a(如果 x 活 10 秒，y 活 5 秒，那么这个 'a 的最大约束范围就是 5 秒)

`-> &'a str` 是在约束返回值的寿命，它的寿命**绝对不能超过** `'a`（即不能超过 `x` 和 `y` 中活得更短的那一个），因为编译器不知道你会返回 `x` 还是 `y`。为了绝对安全，它必须做“**最坏打算**”！

就比如说:
```
fn main() {
    let string1 = String::from("long string is long");
    let result;
    {
        let string2 = String::from("xyz");
        result = longest(string1.as_str(), string2.as_str());
    }
    println!("The longest string is {}", result);//string2已经失效，编译错误
}
```

## 4.深入思考生命周期标注

- 使用生命周期的方式往往取决于函数的功能，例如之前的 `longest` 函数，如果它永远只返回第一个参数 `x`，`y` 完全没有被使用，因此 `y` 的生命周期与 `x` 和返回值的生命周期没有任何关系，意味着我们也不必再为 `y` 标注生命周期，只需要标注 `x` 参数和返回值即可：
```
fn longest<'a>(x: &'a str, y: &str) -> &'a str {
    x
}
```

- **函数的返回值如果是一个引用类型(函数返回的引用，必须指向一个“依然活着”的内存地址)，那么它的生命周期只会来源于**：

1. 函数参数的生命周期

2. 函数体中某个新建引用的生命周期,如：
```
fn longest<'a>(x: &str, y: &str) -> &'a str {
    let result = String::from("really long string");
    result.as_str()
}
```

上面代码的主要问题就在于，`result` 在函数结束后就被释放，但是在函数结束后，对 `result` 的引用依然在继续，解决方法就是转移所有权:
```
fn longest(_x: &str, _y: &str) -> String {
    String::from("really long string")
}

fn main() {
   let result = longest("not", "important");
   println!("{}",result);
}
```

## 5.结构体中的生命周期
简单来说：**如果你在结构体中存储了一个“引用”，你就必须告诉编译器这个引用能存活多久。** 这样可以确保结构体本身不会比它所引用的数据活得更久，从而避免“悬垂指针”错误

语法结构:
```
struct User<'a> {
    username: &'a str, // 这个引用必须至少和结构体活得一样久
    email: String,     // 拥有所有权的数据不需要标注
}
```

- **`&'a str`**: 表示 `username` 字段是一个引用，其生命周期为 `'a`。

错误示范:
```
struct Excerpt<'a> {
    part: &'a str,
}

fn main() {
    let i;
    {
        let novel = String::from("Call me Ishmael.");
        i = Excerpt {
            part: &novel, // 错误！novel 在花括号结束时被释放
        };
    } 
    // 这里 i 依然存在，但它引用的 novel 已经没了
    println!("{}", i.part); 
}
```

核心规则：**结构体所引用的字符串 `str` 生命周期需要大于等于该结构体的生命周期**

## 6.生命周期消除

在 Rust 中，**生命周期消除 是一套编译器自动推导生命周期的规则

有几点需要注意：

- 消除规则不是万能的，若编译器不能确定某件事是正确时，会直接判为不正确，那么你还是需要手动标注生命周期
- **函数或者方法中，参数的生命周期被称为 `输入生命周期`，返回值的生命周期被称为 `输出生命周期`**

#### 三条消除规则

编译器使用三条消除规则来确定哪些场景不需要显式地去标注生命周期。其中第一条规则应用在输入生命周期上，第二、三条应用在输出生命周期上。若编译器发现三条规则都不适用时，就会报错，提示你需要手动标注生命周期

**第一条规则：每个参数都有自己的生命周期**

例如一个引用参数的函数就有一个生命周期标注: `fn foo<'a>(x: &'a i32)`，两个引用参数的有两个生命周期标注:`fn foo<'a, 'b>(x: &'a i32, y: &'b i32)`, 依此类推

**第二条规则：单一输入对应单一输出**

如果只有一个输入生命周期参数，那么该生命周期会被赋给**所有**输出生命周期参数

例如函数 `fn foo(x: &i32) -> &i32`，`x` 参数的生命周期会被自动赋给返回值 `&i32`，因此该函数等同于 `fn foo<'a>(x: &'a i32) -> &'a i32`

**第三条规则：方法中的 `&self` 优先**

如果是一个方法（即第一个参数是 `&self` 或 `&mut self`），那么 `self` 的生命周期会被赋给**所有**输出生命周期参数

我想在这里可以总结一下，**什么时候需要标注生命周期呢**？

  简单来说，当编译器利用以上三条规则却猜不出生命周期时才需要你来标注:
1. 当有多个输入引用，且返回一个引用时
2. 当你想明确指定返回值只和某个特定参数绑定时
3. 定义包含引用的结构体(消除规则只适用于**函数/方法签名**。在定义结构体、枚举或 trait 时，只要里面存了引用，就必须显式标注)

就比如说: `fn longest<'a, 'b>(x: &'a str, y: &'b str) -> &str {...}`//无法编译

## 7.方法中的生命周期

为具有生命周期的结构体实现方法时，我们使用的语法跟泛型参数语法很相似：
```
struct ImportantExcerpt<'a> {
    part: &'a str,
}

impl<'a> ImportantExcerpt<'a> {
    fn level(&self) -> i32 {
        3
    }
}
```

`impl<'a>` 意为:为某个生命周期为'a的参数的类型实现一些方法
其中有几点需要注意的：
- impl 中必须使用结构体的完整名称，包括 <'a>，因为生命周期标注也是结构体类型的一部分！
- 方法签名中，往往不需要标注生命周期，得益于生命周期消除的第一和第三规则:
```
impl<'a> ImportantExcerpt<'a> {
    fn announce_and_return_part(&self, announcement: &str) -> &str {
        println!("Attention please: {}", announcement);
        self.part
    }
}
```
首先，编译器应用第一规则，给予每个输入参数一个生命周期，需要注意的是，编译器不知道 `announcement` 的生命周期到底多长，因此它无法简单的给予它生命周期 `'a`，而是重新声明了一个全新的生命周期 `'b`，接着，编译器应用第三规则，将 `&self` 的生命周期赋给返回值 `&str`

## 8.静态生命周期

在之前我们学过字符串字面量，提到过它是被硬编码进 Rust 的二进制文件中，因此这些字符串变量全部具有 `'static` 的生命周期：`let s: &'static str = "我没啥优点，就是活得久，嘿嘿";

-  生命周期 `'static` 意味着能和程序活得一样久，例如字符串字面量和特征对象
- 实在遇到解决不了的生命周期标注问题，可以尝试 `T: 'static`，有时候它会给你奇迹
- `'static` 是一个**特殊的保留生命周 期关键字**,你不需要像 `'a` 那样在函数名后面声明它

生命周期与泛型同时出现的一个例子（详见10）:
```
use std::fmt::Display;

fn longest_with_an_announcement<'a, T>(
    x: &'a str,
    y: &'a str,
    ann: T,
) -> &'a str
where
    T: Display,
{
    println!("Announcement! {}", ann);
    if x.len() > y.len() {
        x
    } else {
        y
    }
}
```

## 9.深入理解

```
fn print_refs<'a, 'b>(x: &'a i32, y: &'b i32) {
    println!("x is {} and y is {}", x, y);
}

fn failed_borrow<'a>() {
    let _x = 12;
    let y: &'a i32 = &_x;
}

fn main() {
    let (four, nine) = (4, 9);

    print_refs(&four, &nine);
    
    failed_borrow();
```

知道为什么这段代码会报错吗，因为`fail_borrow`这个函数没有参数，但是声明了一个生命周期，此时编译器会将这个函数的生命周期视为全局生命周期，而_x的生命周期是在函数结束后就消失的，y真正的生命周期比声明的小，为了避免悬垂引用，所以编译器会报错！

```
#[derive(Debug)]
struct NoCopyType {}

#[derive(Debug)]
#[allow(dead_code)]
struct Example<'a, 'b> {
    a: &'a u32,
    b: &'b NoCopyType
}

fn fix_me(foo: &Example) -> &NoCopyType
{ foo.b }

fn main()
{
    let no_copy = NoCopyType {};
    let example = Example { a: &1, b: &no_copy };
    fix_me(&example);
    println!("Success!")
}
```

这为什么会报错？错误出在定义的函数`fix_me`这一项，该函数没有声明生命周期，因此编译器会给它一个生命周期`'c` 而在结构体`Example` 中已经给b注明了生命周期`'b`  ，在只有一个输入引用的情况下，它会自动假设**输出引用的生命周期与输入引用的生命周期一致** ，但`'c`与`'b` 可能不一致，所以会报错！

```
#[derive(Debug)]
struct Config {
    a: String,
    b: String,
}
static mut config: Option<&mut Config> = None;//这行代码是在对全局变量 config 进行“静态初始化”

fn init() -> Option<&'static mut Config> {
    let c = Box::new(Config {
        a: "A".to_string(),
        b: "B".to_string(),
    });
    Some(Box::leak(c))
}


fn main() {
    unsafe {
        config = init();

        println!("{:?}",config)
    }
}
```

这段代码写的极为巧妙，`Box::leak` 就像是给编译器开了一个“特许通行证”：它允许你通过**牺牲内存回收控制权**，来换取**最高等级的生命周期权限**

`&'static` 只能说明引用指向的数据是能一直存活的，但是引用本身依然受限于它的作用域：
```
fn main() {
    {
        // 字符串字面量能跟程序活得一样久，因此 `static_string` 的生命周期是 `'static`
        let static_string = "I'm in read-only memory";
        println!("static_string: {}", static_string);

        // 当 `static_string` 超出作用域时，该引用就无法再被使用，但是引用指向的数据( 字符串字面量 ) 依然保存在二进制 binary 所占用的内存中
    }

    println!("static_string reference remains alive: {}", static_string);
}
```

`&'static` 可以被强转成一个较短的生命周期:
```
// 声明一个 static 常量，它拥有 `'static` 生命周期.
static NUM: i32 = 18;

// 返回常量 `Num` 的引用，注意，这里的生命周期从 `'static` 强转为 `'a`
fn coerce_static<'a>(_: &'a i32) -> &'a i32 {
    &NUM
}

fn main() {
    {
        let lifetime_num = 9;

        let coerced_static = coerce_static(&lifetime_num);

        println!("coerced_static: {}", coerced_static);
    }

    println!("NUM: {} stays accessible!", NUM);
}
```

```
use std::fmt::Debug;

fn print_it<T: Debug + 'static>( input: T) {
    println!( "'static value passed in is: {:?}", input );
}

fn print_it1( input: impl Debug + 'static ) {
    println!( "'static value passed in is: {:?}", input );
}


fn print_it2<T: Debug + 'static>( input: &T) {
    println!( "'static value passed in is: {:?}", input );
}
```

这段代码中最值得关注的是`print_it2`中，参数input的类型是借用，它的寿命由调用者T决定，借用的寿命可能不是`'static` ，因此会报错，特征约束`<T: Debug + 'static>` 仅仅只是说明你借用的类型T是满足`’static`的，而你仅仅只是从图书馆短期的借走了一本拥有永久授权的书。

**区别:**
- `T: 'static`：是在检查“类型内部有没有藏着短命的雷”，它不要求引用本身是静态的。
- `&'static T`：是在要求“这个引用指向的数据必须永远不被销毁”。

**“直到程序结束”并不等于 `'static`** !!!

## 10.特征约束

就像泛型类型可以有约束一样，生命周期也可以有约束 ，如下所示：
- T: **'a，所有引用在 T 必须大于等于生命周期 'a**
- T: **Trait + 'a: T 必须实现特征 Trait 并且所有引用在 T 必须大于等于生命周期 'a**
如:
```
/* 使用生命周期注释结构体
1. `r` 和 `s` 必须是不同生命周期
2. `s` 的生命周期需要大于 'r'
*/
struct DoubleRef<'a,'b:'a,T:'b> {
    r: &'a T,
    s: &'b T
}
fn main() {
    println!("Success!")
}
```

```
struct ImportantExcerpt<'a> {
    part: &'a str,
}

impl<'a, 'b> ImportantExcerpt<'a> {
    fn announce_and_return_part(&'a self, announcement: &'b str) -> &'b str {
        println!("Attention please: {}", announcement);
        self.part
    }
}

fn main() {
    println!("Success!")
}
```

 在这段代码中，方法ImportantExcerpt的结果的生命周期是‘a，但是它声明返回的生命周期为’b，编译器不知道‘a与’b的关系，因此拒绝编译，我们要限制‘b要比’a小，才能够顺利编译！

```
fn f<'a:'b, 'b>(x: &'a i32, mut y: &'b i32) {
    y = x;                      
    let r: &'b &'a i32 = &&0;   
}

fn main() {
    println!("Success!")
}
```

为什么要声明’a要大于‘b呢，原因出自`let r: &'b &'a i32 = &&0;` ，这包含一个**生命周期嵌套声明**, &0的生命周期是’a，而&&0的生命周期是‘b,所以’a **outlives** ‘b!

## 11.HRTB（更高等级特征约束）(Higher-ranked trait bounds)

HRTB是一个很抽象的概念，先从实例中认识它：
```
fn main() {
    let my_closure = |name: &str| println!("Hello, {}", name);

    call_with_hello(my_closure);
}

fn call_with_hello<F>(f: F) 
where 
    F: for<'a> Fn(&'a str)// 这就是HRTB！
{
    let local_name = String::from("Rust");
    
    f(&local_name); 
}
```

这段代码写的很巧妙，颠覆了我以往对代码的认识。首先定义了一个闭包函数(**闭包函数就是能捕捉外部变量的匿名函数**)`my_closure` ，值得注意的是，这个函数作为了一个参数传入了`call_with_hello` ，在`call_with_hello`中f就代指`my_closure`函数，而矛盾点就在`local_name`中，它是个在函数中的内部变量，生命周期很短,这时候如果想要编译成功就需要HRTB！

`F: for<'a> Fn(&'a str)`这段代码中，`Fn(&'a str)`意为对函数的接收的字符串引用的生命周期暂时被标记为`'a`，`for<'a>` 意为：对于任何一种可能的生命周期 `'a`。整个的意思是:**对进来的这个函数 F，必须具备处理‘任意寿命’引用的能力!** 

## 12.其他生命周期
#### 1.NLL（非词汇生命周期）(Non-Lexical Lifetime)

什么是NLL？，让我先来看看这样的一段代码:
```
fn main() {
   let mut s = String::from("hello");

    let r1 = &s;
    let r2 = &s;
    println!("{} and {}", r1, r2);

    let r3 = &mut s;
    println!("{}", r3);
}
```

按照以往的知识来说，上面的这段代码一定会违反借用规则而编译失败，但如果能编译成功呢？**编译器在作用域结束之前判断不再使用引用的能力**称为 **非词法生命周期**（简称 **NLL** ）

 **什么是“词法”与“非词法”？**
- **词法生命周期 (Old Rust)**：生命周期是由代码的**大括号 `{ }`（作用域）**决定的。只要变量在括号里，它就一直占着位置，即使它后面根本没被用到。
- **非词法生命周期 (Modern Rust/NLL)**：生命周期是由**代码的实际执行路径**决定的。一旦一个变量以后再也不被用到了，它的生命周期就提前结束，哪怕还没到大括号。

简单来说就是, 非词汇生命周期就是:**一个引用的寿命，结束于它最后一次被使用的地方，而不是它所在的大括号终点**。

#### 2.再借用

简单来说：**当你把一个“可变引用”传递给一个函数或者赋值给另一个变量时，Rust 并不总是“移动”它，而是悄悄地创建了一个新的“子借用”**

在 Rust 中，**可变引用（`&mut T`）是不允许拷贝（Copy）的，只能移动（Move）**。 按理说，如果你把一个可变引用传给了一个函数，原本的变量就应该失效了。但实际情况是：
```
fn main() {
    let mut count = 10;
    let r = &mut count; // r 是一个可变引用

    // 第一次调用函数
    do_something(r); 

    // 照理说 r 已经被“移动”进函数了，为什么这里还能接着用？
    do_something(r); // 依然合法！
}

fn do_something(val: &mut i32) {
    *val += 1;
}
```
**原因就是：** 在执行 `do_something(r)` 时，编译器并没有把 `r` 整个“搬走”，而是以 `r` 为基础，**再借了一份**临时的可变引用。

**显式再借用**：
编译器通常会自动帮你处理再借用，但你也可以手动写出来，它的语法是 `&mut *r`：
```
let mut x = 5;
let r = &mut x;

let rr = &mut *r; // 手动再借用
// 注意：此时 r 被冻结，不能使用，直到 rr 离开作用域（或根据 NLL 规则不再使用）

*rr += 1; 
// println!("{}", r); // 如果取消注释这行，r 的解冻会发生在这里之后
```

- `*r`：解引用。意思是“顺着 `r` 找到它背后的那个 `x`”。
- `&mut ...`：再次对这个 `x` 进行可变借用。

Rust 规定：**同一时间只能有一个活跃的可变权限** 因此在rr发挥完作用前，r都会被冻结
